## 왜 이 글을 찾아봤나

- **계기:** 회사 팀에서 Skill을 어떻게 함께 쌓을지 논의가 나왔다.
- **당시 상황:** 그 대화 전에, Skill을 잘 만드는 법을 정리한 글을 먼저 읽어보고 싶었다.
- **궁금했던 질문:** 문법 말고 실무에서 "좋은 Skill"이 설계적으로 무엇인지.

---

## 읽으면서 느낀 점

- **인상 깊었던 부분:** Skill을 잘 만든다는 건 프롬프트 꾸미기보다 일하는 방법을 설계하는 일에 가깝다. 추상화 기반 프로그래밍처럼, 어떤 층을 드러내고 무엇을 숨길지 정하는 일이기도 했다.
- **의아했거나 동의하기 어려웠던 부분:** 문서 작성과 맥락이 비슷하다는 설명이 처음엔 과한 비유처럼 느껴졌다. 구조화·운영까지 묶어 읽으니 시스템 설계와 결이 맞는다고 납득했다.
- **내 경험과 연결지어 보면:** 코드·문서에서 쓰던 감각—추상화, 점진적 공개, 명확한 "언제 쓰는지"—이 팀 Skill을 쌓는 논의에도 그대로 통한다.

---

## 배운 것

### 핵심 takeaway (3줄 요약)

1. Skill은 화려한 프롬프트가 아니라 작은 규모의 업무 방식 설계다.
2. 추상화·점진적 공개가 핵심이다. 코드나 문서에서 쓰던 감각이 그대로 통한다.
3. 팀이 공유할수록 구조와 운영(거버넌스)이 더 중요해진다.

### 글에서 짚고 싶은 개념

| 주제 | 내가 이해한 바 |
|------|----------------|
| 간결함 — 컨텍스트 윈도우는 공공재 | 한 줄 한 줄이 주의를 나눠 가진다. 과감히 줄인다. |
| 적절한 자유도 (높음 / 중간 / 낮음) | 작업이 얼마나 깨지기 쉬운지에 맞춰 지시의 단단함을 조절한다. |
| Progressive disclosure (점진적 공개) | 개요를 먼저 보여 주고, 깊이는 필요할 때만 연다. |
| SKILL.md 구조 (frontmatter, when to use) | frontmatter와 "언제 쓰는지"는 일종의 API 계약이다. |
| 테스트 & 반복 개선 | 실제 사용에서 빈틈이 보이면 코드·문서처럼 고친다. |

### 바로 적용해볼 수 있는 것

- [ ] 팀 Skill 논의에 설계·추상화·문서화 프레이밍을 가져가 본다.
- [ ] "when to use"와 얇은 해피 패스가 있는 팀 Skill 초안을 하나 써 본다.

---

## 메모

팀 단위 Skill 축적 이야기를 하기 전에 설계 언어를 맞추는 데 도움이 됐다. 체크리스트를 베낄 글이라기보다, 일을 어떻게 구조화할지 보는 렌즈에 가깝다.
