## 왜 이 글을 찾아봤나

- **계기:** 회사 팀에서 Skill을 어떻게 함께 쌓을지 논의가 나왔다.
- **당시 상황:** 그 대화에 앞서, Skill을 잘 만드는 방법에 대해 잘 다듬어진 글을 먼저 읽어보고 싶었다.
- **궁금했던 질문:** 문법이 아니라, 실무에서 “좋은 Skill”이란 설계적으로 무엇을 의미하는지.

---

## 읽으면서 느낀 점

- **인상 깊었던 부분:** Skill을 잘 만든다는 건 프롬프트를 꾸미는 일이 아니라, **일하는 방법을 설계하는 것**에 가깝다고 느꼈다.
- **의아했거나 동의하기 어려웠던 부분:** **추상화 기반 프로그래밍**과 감이 비슷했다—어떤 층을 노출하고 무엇을 숨길지, 에이전트에 필요한 것만 드러내는 식으로.
- **내 경험과 연결지어 보면:** **문서를 잘 쓰고 관리하는 것**과도 맥락이 닮았다. 결국 문제 해결을 위한 방법을 **잘 구조화해서 작성하는 것**이 중요하고, 일종의 **시스템 설계**와 결이 맞는다.

---

## 배운 것

### 핵심 takeaway (3줄 요약)

1. Skill은 화려한 프롬프트가 아니라, 작은 단위의 **업무 방식 설계**다.
2. **추상화·점진적 공개**가 핵심이다—코드나 문서에서 쓰는 감각이 그대로 통한다.
3. 팀이 공유할수록 **구조와 운영(거버넌스)** 관점이 더 중요해진다.

### 글에서 짚고 싶은 개념

| 주제 | 내가 이해한 바 |
|------|----------------|
| 간결함 — 컨텍스트 윈도우는 공공재 | 한 줄 한 줄이 주의를 나눠 가진다. 과감히 줄인다. |
| 적절한 자유도 (높음 / 중간 / 낮음) | 작업이 얼마나 깨지기 쉬운지에 맞춰 지시의 단단함을 조절한다. |
| Progressive disclosure (점진적 공개) | 개요 먼저, 깊이는 필요할 때만 연다. |
| SKILL.md 구조 (frontmatter, when to use) | frontmatter와 “언제 쓰는지”는 일종의 API 계약이다. |
| 테스트 & 반복 개선 | 실제 사용에서 빈틈이 보이면 코드·문서처럼 고친다. |

### 바로 적용해볼 수 있는 것

- [ ] 팀 Skill 논의에 이 프레이밍(설계·추상화·문서화)을 가져가기.
- [ ] “when to use”와 얇은 해피 패스가 있는 팀 Skill 초안 하나 써보기.

---

## 메모

팀 단위 Skill 축적 이야기를 하기 전에, **설계 언어**를 맞추는 데 도움이 됐다. 그대로 베낄 체크리스트라기보다, 우리가 일을 어떻게 구조화할지 보는 렌즈에 가깝다.
