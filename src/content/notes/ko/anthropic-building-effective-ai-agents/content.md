## 왜 이 글을 찾아봤나

- **계기:** Anthropic 엔터프라이즈용 AI 에이전트 아키텍처 가이드.
- **당시 상황:** 에이전트 설계 관점을 다시 정리하고 싶었다. 단일 vs 멀티 에이전트, 워크플로, 복잡도를 언제 올릴지.
- **궁금했던 질문:** 프로덕션 에이전트에 대해 Anthropic이 권하는 프레이밍이 무엇인지, 내가 가정해 온 것과 맞는지.

---

## 읽으면서 느낀 점

- **인상 깊었던 부분:** 크게 새로운 내용은 없었다. 단순하게 시작하고, 측정 가능한 가치가 있을 때만 복잡도를 올리라는 이야기를 깔끔하게 정리한 글.
- **의아했거나 동의하기 어려웠던 부분:** 고객 사례 수치는 인상적이지만 마케팅 느낌이 강하다. 실질적으로 쓸 만한 건 아키텍처 가이드 쪽.
- **내 경험과 연결지어 보면:** 마인드셋 리프레시 정도. 멀티 에이전트로 바로 가지 말라는 걸 다시 확인했다.

---

## 배운 것

### 핵심 takeaway (3줄 요약)

1. **생성형 AI는 질문에 답하고, 에이전트는 문제를 푼다.** 작업을 분석하고 도구를 고르고, 시도·평가·수정을 반복한다. 스크립트 자동화와 다르다.
2. **단일 에이전트부터 시작한다.** 비용·디버깅·지표 측면에서 유리하다. 멀티 에이전트는 토큰을 10~15배 쓴다. 비즈니스 가치가 비용을 정당화할 때만.
3. **기술적 화려함이 아니라 제약에 맞춰 아키텍처를 고른다.** 통제 수준, 문제 복잡도, 예산, 도메인 깊이.

### 가이드 요약

| 영역 | 핵심 |
|------|------|
| 설계 원칙 | 작업에 맞는 모델; 모듈러 설계(프롬프트·도구·에이전트); Agent Skills로 도메인 전문성; 첫날부터 관찰 가능성 |
| 단일 에이전트 | 인지 → 계획 → 실행 → 관찰 루프. 경로가 미리 정해지지 않은 개방형 문제에 적합 |
| 멀티 에이전트 | 계층형(슈퍼바이저 위임) 또는 협업형(P2P). Anthropic 연구 기준 복잡 병렬 작업에서 ~90% 향상, 토큰 비용 높음 |
| 워크플로 | 순차(감사 추적 용이), 병렬(독립 분석), 평가-최적화(생성 → 검토 → 수정 반복) |
| 선택 프레임워크 | 통제 높음 → 단일/순차; 중간 → 계층형; 낮음 → 협업형. 단일 도메인 → 에이전트 + Skills; 다도메인 협업 → 멀티 에이전트 |
| 진화 경로 | 단일로 ROI 검증 → 라우팅 → 전문화 → 연동 → 품질 평가. 데이터로 가치가 확인될 때만 복잡도 추가 |

### 프로덕션 사례 (발췌)

- Coinbase: Claude 고객 지원, 시간당 수천 건, 99.99% 가용성
- Intercom Fin: 최대 86% 해결률
- 은행 신용 리스크 메모: 생산성 20~60% 향상, 처리 30% 단축
- Inscribe(사기 탐지): 검토 30분 → 90초

### 바로 적용해볼 수 있는 것

- 없음. 마인드셋 리프레시였고, 이어서 파볼 주제도 딱히 없다.
